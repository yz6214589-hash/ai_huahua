## 一、数据来源说明

### 1.1 数据来源概述

基本面选股模块当前的数据来源存在两个层次：**后端数据库存储的真实数据**和**前端展示层的模拟数据**。这两个层次的数据来源、更新频率和数据完整性存在显著差异，需要明确区分。

**后端真实数据**通过定时任务从东方财富（akshare）接口采集，存储在MySQL数据库中。前端展示层当前使用的是预定义的模拟数据（MOCK_RESULTS），并未调用后端API进行真实数据查询。这种设计导致前端展示的选股结果无法反映市场的真实情况，需要后续优化。

### 1.2 后端数据采集接口

后端财务数据采集使用akshare库的`stock_financial_analysis_indicator_em`函数，该函数从东方财富获取A股上市公司的财务指标数据。具体的数据采集逻辑位于文件`/Users/apple/Desktop/ai_huahua/ai_quant/backend/core/jobs/domains/stock_financial.py`中的`run_stock_financial`函数。

股票列表的获取使用akshare库的`stock_zh_a_spot_em`函数，该函数返回A股全市场股票列表，包括上海证券交易所和深圳证券交易所的所有A股股票。代码中通过判断股票代码前缀（6开头为上海，其他为深圳）自动推断交易所信息。

数据采集任务支持两种运行模式：测试模式（test）和全量模式。全量模式下默认每次处理50只股票，可通过参数`max_stocks`配置。每次采集每个股票的最近12期财务报告数据，可通过参数`max_rows_per_stock`调整。测试模式默认只处理单只股票（贵州茅台600519.SH），便于开发调试。

### 1.3 数据存储介质

所有基本面相关数据存储在MySQL数据库中，数据库配置通过`core.db`模块管理。主要涉及以下数据表：

**trade_stock_financial表**存储股票财务数据，是基本面选股的核心数据表。该表的结构包含股票代码（stock_code）、报告期（report_date）、营业总收入（revenue）、净利润（net_profit）、每股收益（eps）、净资产收益率（roe）、总资产净利率（roa）、毛利率（gross_margin）、净利率（net_margin）、资产负债率（debt_ratio）、流动比率（current_ratio）、经营活动现金流（operating_cashflow）、总资产（total_assets）、总权益（total_equity）以及数据来源标识（data_source）。

**trade_stock_master表**存储股票基础信息，包含股票代码（stock_code）、股票名称（stock_name）、数据来源（source）、申万一级行业（sector_level1）、申万二级行业（sector_level2）、所属市场（market）、上市日期（list_date）、行业基准指数（industry_benchmark）、是否ST股（is_st）和股票状态（status）等字段。这个表提供了行业分类信息，是实现行业筛选功能的基础。

**trade_stock_daily表**存储股票日线行情数据，包括交易日期（trade_date）、收盘价（close_price）、成交量（volume）、RSI14指标（rsi14）、MA20均线（ma20）和股票名称（stock_name）。该表在扩展后还包含申万行业分类、板块相位、动量因子、Alpha得分等字段。

### 1.4 数据更新频率

数据采集任务通过jobs系统的调度机制执行，支持定时触发。默认情况下，财务数据按季度更新（因为财务报告按季度披露），但可以通过配置cron表达式实现更频繁的采集。实际部署中建议将财务数据采集任务的执行周期设置为每周一次或每日一次，以保证数据的时效性。

日线行情数据的采集频率可以设置为每日收盘后执行，确保获取最新的交易数据。系统支持增量更新机制，只采集自上次采集日期之后的新数据，避免重复写入。

### 1.5 数据覆盖范围

当前数据覆盖范围包括：**A股全市场**股票，涵盖上海证券交易所（以6开头的股票代码）和深圳证券交易所（以0、3开头的股票代码）的所有A股。数据不包括港股、美股等其他市场的股票。

### 1.6 数据校验机制

数据采集过程中实现了多层校验机制。在数据源层面，使用akshare库的异常捕获机制，当单只股票数据获取失败时记录失败列表并继续处理其他股票，不会因为单只股票失败而导致整个任务中断。

在数据转换层面，使用`safe_float`函数进行类型安全转换，将None、空字符串等异常值转换为None，同时过滤掉无穷大值（float("inf")和float("-inf")），确保数据质量。

在数据库写入层面，使用`ON DUPLICATE KEY UPDATE`语法实现Upsert逻辑，当记录已存在时更新字段值，避免重复插入。同时使用`COALESCE`函数确保NULL值不会覆盖已存在的有效数据。

### 1.7 数据缺失与范围遗漏问题

经过分析，当前模块存在以下数据缺失问题需要关注：

**市值数据缺失**：前端筛选条件包含“总市值（亿）”字段，但后端数据库中trade_stock_financial表和trade_stock_master表均未存储市值字段。akshare的`stock_zh_a_spot_em`函数返回的数据包含总市值和流通市值信息，但当前的数据采集任务（stock_financial.py）并未提取和存储这些字段。

**估值指标缺失**：市盈率（PE）和市净率（PB）是前端筛选的核心指标，但当前财务数据表中只有`pe_ttm`和`pb`字段定义，未在实际采集过程中填充数据。PE和PB需要通过实时行情数据计算或从专门的估值数据接口获取。

**增长指标缺失**：营收增速（rev_growth）和利润增速（profit_growth）是重要的筛选条件，当前数据表中虽然有`revenue_growth_yoy`和`profit_growth_yoy`字段，但采集任务未实现同比增速的计算逻辑。增速数据需要通过对比相邻年度或季度的财务数据计算得出。

**行业映射不完整**：当前使用前端定义的简化行业分类（bank、insurance、securities等12个分类），而后端使用的是申万行业分类（一级31个、二级134个）。两套分类体系未建立映射关系，导致前端筛选无法准确匹配后端数据。

## 二、业务逻辑说明

### 2.1 业务流程概述

基本面选股功能的完整业务流程包含以下主要环节：**用户输入阶段**、**条件解析阶段**、**数据检索阶段**、**筛选匹配阶段**、**结果计算阶段**和**结果返回阶段**。当前实现中，部分环节在前端完成，部分环节在后端完成，整体链路尚未完全打通。

**用户输入阶段**：用户在基本面选股页面通过滑块和复选框设置筛选条件。支持的筛选条件包括市盈率范围（0-200，默认0-30）、市净率范围（0-20，默认0-10）、ROE范围（-50%-100%，默认0%-100%）、毛利率范围（0-100%，默认0-100%）、净利率范围（-50%-100%，默认0-100%）、行业板块（多选，支持12个行业分类）、总市值范围（0-50000亿，默认0-50000）、营收增速范围（-100%-500%，默认-100%-500%）、利润增速范围（-500%-1000%，默认-500%-1000%）和资产负债率范围（0-100%，默认0-80%）。

**条件解析阶段**：前端将用户设置的筛选条件转换为内部数据结构`FilterValues`，每个筛选条件包含最小值（min）、最大值（max）和选中选项（selected）三个属性。对于范围型筛选，使用滑块的双端值；对于多选型筛选，使用选中的选项列表。

**数据检索阶段**：当前实现中，前端直接使用预定义的MOCK_RESULTS模拟数据进行展示，并未向后端发起真实的数据查询请求。这意味着所有筛选和计算都在浏览器端执行，无法获取全市场股票的真实数据。

**筛选匹配阶段**：前端通过`matches`函数对模拟数据进行过滤，逐一检查每只股票是否满足所有筛选条件。对于范围型条件，使用大于等于最小值且小于等于最大值的判断逻辑；对于多选条件，如果未选择任何选项则通过检查，否则要求股票行业必须在选中列表中。

**结果计算阶段**：对通过筛选的股票计算综合评分。评分公式为：`score = (pe_ok ? 20 : 0) + (roe_ok ? 30 : 0) + (profit_ok ? 25 : 0) + (s.gross / 100 * 15) + (s.net / 100 * 10)`。其中pe_ok表示市盈率在合理范围（0到设定最大值之间）且大于0，roe_ok表示ROE大于等于设定最小值，profit_ok表示利润增速大于等于设定最小值。

**结果返回阶段**：将筛选结果和评分以表格形式展示给用户，包含股票代码、名称、行业、市盈率、市净率、ROE、毛利率、净利率、市值、营收增速、利润增速、负债率和综合评分等列。

### 2.2 筛选条件生效逻辑

各筛选条件的生效逻辑遵循以下规则：

**范围型条件**：当用户拖动滑块设置最小值和最大值时，满足条件的股票必须使目标字段的值在[min, max]区间内。对于负数支持的字段（如ROE、净利率），允许设置负值范围。特殊地，市盈率字段在显示时对负值（亏损）股票显示为“亏损”文本，不参与数值范围匹配。

**多选型条件**：行业板块筛选支持多选，用户可以同时选择多个行业。当至少选择一个行业时，只有属于被选中行业的股票才能通过筛选；当未选择任何行业时，该条件不生效，所有股票都可以通过。

**条件组合**：多个筛选条件之间是“AND”关系，即股票必须同时满足所有已设置的筛选条件才能出现在结果中。条件之间无优先级区分，按FILTERS数组定义的顺序逐一检查。

### 2.3 多条件组合运算规则

多条件组合采用严格的“与运算”策略，代码实现逻辑为：

```typescript
for (const f of FILTERS) {
  const v = values[f.key];
  if (f.type === 'range') {
    const val = (s as Record<string, unknown>)[f.key];
    const num = typeof val === 'number' ? val : 0;
    if (num < v.min || num > v.max) return false;
  } else if (f.type === 'multiselect') {
    if (v.selected.length === 0) continue;
    const ind = s.industry;
    if (!v.selected.includes(ind)) return false;
  }
}
return true;
```

该逻辑的特点是：任何一项条件不满足立即返回false（短路求值），只有通过所有条件检查的股票才会被纳入结果集。这种设计保证了筛选结果的可预测性，但也意味着如果多个条件设置过于严格，可能导致结果为空。

### 2.4 股票池准入标准

当前模块没有明确的股票池准入标准限制，所有A股股票都在候选范围内。但在实际业务中，建议建立以下准入门槛：

**财务数据完整性**：股票必须至少有一期有效的财务报告数据才能参与筛选。缺失财务数据的股票应被排除或标记为数据不可用。

**上市时间要求**：建议设置最低上市时间要求（如上市满一年），以确保有足够的历史财务数据进行基本面分析。

**交易状态过滤**：应排除退市、停牌等非正常交易状态的股票，这些股票的基本面筛选结果不具有实际参考价值。

**ST股票处理**：根据业务需求决定是否纳入ST股票。ST股票可能存在财务异常，其财务指标筛选结果可能失真。

### 2.5 异常场景业务处理

当前模块对异常场景的处理如下：

**数据缺失处理**：模拟数据中各字段均有值，不存在缺失情况。真实数据场景下，需要处理财务数据缺失的情况，建议将缺失值视为不满足筛选条件（排除）或使用行业平均值填充。

**结果为空处理**：当没有任何股票满足筛选条件时，前端显示“暂无符合条件的股票”提示信息，引导用户调整筛选条件。

**计算异常处理**：评分计算中涉及除法运算（s.gross / 100和s.net / 100），当gross或net为None时会被转换为0，不会导致计算错误。PE、ROE、利润增速的判断条件中对负值和0值有特殊处理逻辑。

## 三、代码逻辑与函数说明

### 3.1 前端核心函数

#### 3.1.1 FilterDef接口定义

```typescript
interface FilterDef {
  key: string;           // 筛选条件字段标识
  label: string;         // 界面显示标签
  type: 'range' | 'select' | 'multiselect' | 'number'; // 筛选类型
  min?: number;          // 数值型筛选的最小值
  max?: number;          // 数值型筛选的最大值
  step?: number;         // 滑块步进值
  options?: { value: string; label: string }[]; // 多选选项列表
  defaultMin?: number;   // 默认最小值
  defaultMax?: number;   // 默认最大值
}
```

**功能说明**：定义筛选条件的元数据结构，包括筛选类型、取值范围、默认值等配置信息。

**依赖关系**：被`initValues`函数使用，根据FilterDef配置初始化筛选值状态。

#### 3.1.2 initValues函数

```typescript
const initValues = (): FilterValues => {
  const v: FilterValues = {};
  for (const f of FILTERS) {
    if (f.type === 'range') {
      v[f.key] = {
        min: f.defaultMin ?? f.min ?? 0,
        max: f.defaultMax ?? f.max ?? 100,
        selected: []
      };
    } else {
      v[f.key] = { min: 0, max: 100, selected: [] };
    }
  }
  return v;
};
```

**入参定义**：无显式入参，使用全局常量FILTERS。

**出参结构**：返回`FilterValues`类型对象，键为筛选条件标识，值为包含min、max和selected的对象。

**核心逻辑**：遍历所有筛选条件定义，根据类型初始化状态值。范围型使用配置的默认值，多选型初始化为空选中列表。

**依赖关系**：依赖FILTERS全局常量，被`StockSelectFundamental`组件在初始化时调用。

#### 3.1.3 matches筛选函数

```typescript
const matches = MOCK_RESULTS.filter((s) => {
  for (const f of FILTERS) {
    const v = values[f.key];

      const val = (s as Record<string, unknown>)[f.key];
      const num = typeof val === 'number' ? val : 0;

```

**入参定义**：无显式入参，依赖闭包中的values状态和MOCK_RESULTS数据。

**出参结构**：返回`typeof MOCK_RESULTS[0][]`类型数组，包含所有满足条件的股票记录。

**核心逻辑**：对MOCK_RESULTS中的每条记录逐一检查所有筛选条件，使用短路求值策略，任一条件不满足立即返回false。

**边界条件**：多选条件selected为空数组时跳过检查视为通过；数值字段为非number类型时默认按0处理。

**调用关系**：被组件在渲染阶段调用，用于计算符合条件的股票数量和展示列表。

#### 3.2.2 _get_stock_list函数

```python
def _get_stock_list(test_mode: bool, test_stock: str, max_stocks: int) -> list[str]:
    if test_mode:
        s = normalize_stock_code(test_stock)
        return [s] if s else []
    import akshare as ak

    df = ak.stock_zh_a_spot_em()
    if df is None or len(df) == 0:
        return []
    out: list[str] = []
    for _, r in df.iterrows():
        code_num = str(r.get("代码") or "").strip()
        if not code_num:
            continue
        out.append(f"{code_num}.{_infer_exchange(code_num)}")
        if 0 < max_stocks <= len(out):
            break
    return out
```

**入参定义**：

- test_mode: bool，必填，是否为测试模式
- test_stock: str，必填，测试模式下的股票代码
- max_stocks: int，必填，最大股票数量限制

**出参结构**：返回格式为"代码.交易所"的股票代码列表，如["600519.SH", "000858.SZ"]。

**核心逻辑**：测试模式返回单只股票；全量模式调用akshare获取A股全市场列表，按配置限制返回数量。

**边界条件**：akshare接口返回空DataFrame时返回空列表；股票代码为空时跳过。

#### 3.2.3 _infer_exchange函数

```python
def _infer_exchange(code_num: str) -> str:
    if code_num.startswith("6"):
        return "SH"
    return "SZ"
```

**入参定义**：code_num: str，必填，股票代码数字部分。

**出参结构**：返回"SH"（上海）或"SZ"（深圳）。

**核心逻辑**：根据股票代码前缀推断交易所，6开头为上海交易所，其余为深圳交易所。

**边界条件**：此逻辑不包含北京证券交易所（以8开头）的股票，需要扩展。

### 3.4 代码调用链路

基本面选股模块的完整调用链路如下：

**前端展示链路**：用户访问StockSelectFundamental页面 → React组件初始化调用initValues → 用户调整筛选条件触发状态更新 → matches函数执行筛选逻辑 → ScoreTag计算评分 → 渲染结果表格。

**后端数据链路**：Jobs调度触发run_stock_financial任务 → _get_stock_list获取股票列表 → 遍历调用akshare获取财务数据 → _INSERT_SQL批量写入数据库。

当前前端和后端的调用链路是断开的，前端使用MOCK_RESULTS模拟数据，未接入后端API。这是最关键的待优化项。

## 四、潜在问题排查与优化建议

### 4.1 当前功能存在的潜在问题

#### 4.1.1 数据链路断裂问题

**问题描述**：前端基本面选股页面使用预定义的模拟数据（MOCK_RESULTS）进行展示，并未调用后端API获取真实市场数据。这意味着用户在页面上看到的选股结果是固定的示例数据，而非根据其筛选条件在全市场股票中实时筛选的结果。

**影响范围**：所有使用该页面的用户都无法获得真实的选股结果，筛选功能形同虚设。

**排查步骤**：检查前端代码中是否存在对后端API的调用；检查是否存在类似`/api/v1/stock/select/fundamental`的选股专用接口。

**优先级**：P0（最高）

#### 4.1.2 市值数据缺失问题

**问题描述**：前端筛选条件包含“总市值（亿）”字段，但后端财务数据表（trade_stock_financial）中未存储市值相关字段。akshare的`stock_zh_a_spot_em`函数返回的数据包含总市值和流通市值，但采集任务未提取这些字段。

**影响范围**：市值筛选功能无法实现，用户无法按市值范围筛选股票。

**排查步骤**：检查stock_financial.py中的数据映射代码是否包含market_cap字段；检查trade_stock_financial表结构是否包含市值字段。

**优先级**：P1（高）

#### 4.1.3 增长指标缺失问题

**问题描述**：前端筛选条件包含“营收增速”和“利润增速”，但当前财务数据表中的`revenue_growth_yoy`和`profit_growth_yoy`字段未在实际采集过程中填充数据。这些字段虽然在schema中定义，但采集代码只提取了原始财务指标，未计算同比增长率。

**影响范围**：增长类筛选条件无法实现，用户无法按营收或利润增速筛选股票。

**排查步骤**：检查采集代码是否实现了增速计算逻辑；检查历史数据是否需要回溯计算。

**优先级**：P1（高）

#### 4.1.4 行业分类不一致问题

**问题描述**：前端使用简化的12个行业分类（bank、insurance、securities等），而后端使用的是申万行业分类（一级31个、二级134个）。两套分类体系未建立映射关系，导致前端筛选无法准确匹配后端数据。

**影响范围**：行业筛选功能无法准确工作，用户选择的行业可能与实际股票所属行业不匹配。

**排查步骤**：检查trade_stock_master表中的sector_level1/sector_level2字段是否已填充数据；检查前端行业选项是否需要重新设计。

**优先级**：P2（中）

#### 4.1.5 PE/PB估值指标缺失问题

**问题描述**：前端筛选条件包含市盈率（PE）和市净率（PB），但这两个指标需要通过实时行情数据计算或从专门的数据接口获取。当前系统只存储了字段定义，未实际采集或计算这些指标。

**影响范围**：PE和PB筛选功能无法实现，这两个最重要的估值指标无法使用。

**排查步骤**：检查是否有专门的估值数据采集任务；评估是否需要对接外部估值数据源或通过计算获取。

**优先级**：P1（高）

### 4.2 代码可优化方向建议

#### 4.2.1 建立完整的选股API

**建议内容**：开发专门的`/api/v1/stock/select/fundamental`接口，支持多条件组合查询，返回满足条件的股票列表和评分结果。

**实现要点**：

- 设计灵活的查询参数结构，支持所有筛选条件的任意组合
- 优化SQL查询语句，使用索引加速多条件筛选
- 实现分页返回，避免返回过多结果导致性能问题
- 考虑使用缓存机制，减少重复查询

**预期收益**：打通前后端数据链路，使用户能够基于真实数据进行选股筛选。

#### 4.2.2 增强数据采集范围

**建议内容**：扩展stock_financial采集任务，增加以下字段的采集和计算：

- 总市值和流通市值：从akshare的`stock_zh_a_spot_em`提取
- 同比增速计算：基于历史财务数据计算营收和净利润的同比增长率
- PE和PB估值：从实时行情数据计算或对接专门的估值数据源

**实现要点**：

- 评估akshare是否提供所需数据，确定数据源
- 设计增量计算逻辑，避免每次全量计算
- 历史数据回溯计算，确保数据完整性

**预期收益**：完善数据覆盖，使所有筛选条件都能正常工作。

#### 4.2.3 统一行业分类体系

**建议内容**：建立前端行业分类与后端申万行业分类的映射关系，或者重新设计前端的行业筛选逻辑，与后端分类体系保持一致。

**实现要点**：

- 建立映射表，将前端的12个简化分类映射到申万行业
- 或扩展前端筛选选项，使用完整的申万行业分类
- 确保行业数据的采集和更新与股票列表同步

**预期收益**：行业筛选功能准确工作，用户能够按真实行业分类筛选股票。

#### 4.2.4 优化评分算法

**建议内容**：当前的评分算法是简单加权求和，权重固定且缺乏客观依据。建议引入更科学的评分体系。

**改进方向**：

- 基于行业平均值的相对评分，消除行业间差异
- 引入因子模型（如Fama-French）计算Alpha得分
- 允许用户自定义各指标的权重
- 增加评分的时间维度，如考虑增长趋势

**预期收益**：评分结果更具参考价值，能够真正反映股票的基本面质量。

#### 4.2.5 增加数据校验和监控

**建议内容**：在数据采集和查询各环节增加数据质量校验，建立数据监控机制。

**实现要点**：

- 采集阶段：校验数据完整性、合理性，过滤异常值
- 存储阶段：记录数据更新时间戳，标记数据新鲜度
- 查询阶段：向用户展示数据的时间范围和完整度提示
- 监控告警：数据采集失败、延迟时发送告警通知

**预期收益**：提升数据可信度，快速发现和定位数据问题。

#### 4.2.6 性能优化

**建议内容**：针对全市场股票筛选的性能问题进行优化。

**优化方向**：

- 为常用查询字段建立复合索引（如sector_level1 + roe + debt_ratio）

- 考虑使用列式存储数据库（如ClickHouse）加速分析查询
