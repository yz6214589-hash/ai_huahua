# 智能研报模块代码逻辑映射文档

> 版本：v2.0 | 更新日期：2026-05-12 | 维护人：AI Quant Team

---

## 一、模块概览

智能研报模块负责接收用户输入的股票代码，通过 RAG 检索增强 + LLM 生成，输出结构化 Markdown 研报。模块分为三层：

| 层级 | 文件 | 职责 |
|------|------|------|
| **API 路由层** | `api/reports.py` | HTTP 接口、任务分发、Worker 调度 |
| **RAG 服务层** | `infra/reports/rag.py` | PDF 解析、向量化、FAISS 索引构建与检索 |
| **任务持久化层** | `infra/storage/report_store.py` | 任务记录（MySQL + 文件双写 + Worker Queue） |

## 二、任务持久化层：`infra/storage/report_store.py`

### 2.1 核心数据模型

**`ReportTaskRecord` 数据类**（行 29-54）
```python
@dataclass
class ReportTaskRecord:
    task_id       : str       # UUID hex，任务唯一标识
    model         : str       # "qwen-max" | "deepseek"
    stock_codes   : list[str] # 股票代码列表
    stock_names  : list[str] # 股票名称列表
    use_rag      : bool      # 是否启用 RAG
    status       : str       # waiting/running/success/failed
    created_at   : str       # ISO 时间戳
    started_at   : str | None
    finished_at  : str | None
    error_message: str | None
    error_location: str | None
    report_path  : str | None
    report_markdown: str | None
```

### 2.2 任务持久化核心逻辑

**写入流程（三重保险）**：

```
用户请求 → 内存 dict (_TASKS)
           → JSON 文件 (.ai_quant/report_tasks/{task_id}.json)  原子写入（先写 .tmp 再 rename）
           → MySQL (ai_quant_report_tasks) UPSERT（ON DUPLICATE KEY UPDATE）
```

**`create_task`**（行 556-585）：生成 UUID → 构造 `ReportTaskRecord(status="waiting")` → `_persist()` 双写
```python
task_id = uuid4().hex                      # 行 570
with _LOCK:                              # 行 581 线程安全
    _TASKS[task_id] = rec               # 行 583 写内存
    _persist(rec)                        # 行 584 写磁盘 + MySQL
```

**`_persist`**（行 521-543）：原子写入
```python
tmp = task_path.with_name(f".{task_id}.json.tmp")
tmp.write_text(json, encoding="utf-8")   # 行 538 先写临时文件
tmp.replace(task_path)                    # 行 540 原子替换为正式文件
```

**`_mysql_upsert`**（行 188-257）：UPSERT 到 MySQL（若启用）
```python
INSERT INTO ai_quant_report_tasks ... ON DUPLICATE KEY UPDATE ...
```

**`update_task`**（行 588-609）：更新任务字段 → `_persist` 重新持久化

### 2.3 引导加载（Bootstrap）

**`_load_once`**（行 286-341）：进程启动时一次性从磁盘恢复任务
- 扫描 `.ai_quant/report_tasks/*.json`
- 解析文件名作为 task_id
- 构造 `ReportTaskRecord` 存入内存 dict

### 2.4 存储目录规则

| 环境变量 | 默认值 | 用途 |
|---------|--------|------|
| `AI_QUANT_REPORT_TASK_STORE_DIR` | `.ai_quant/report_tasks/` | 任务 JSON 文件存储 |
| `AI_QUANT_REPORT_MYSQL_ENABLED` | `"1"` | 是否写 MySQL |
| `AI_QUANT_REPORT_STORE_BOOTSTRAP` | `"1"` | 是否从文件/日志引导恢复 |

### 3.2 任务创建流程（完整链路）

**Step 1 - 入队**：`reports_create_task`（行 839-866）

```
1. 参数校验
   └─ 行 857-860：model 必须是 qwen-max/deepseek
   └─ 行 859-860：stock_codes 不能为空
2. 解析股票名称
   └─ _resolve_stock_names(stock_codes) 行 117-151
       ├─ 优先调用 rag.py:resolve_stock_name_by_code() 从 SQLite documents 表查名称
       └─ 未命中则调 core.data:search_stocks() 查询
3. 创建任务记录
   └─ report_store.py:create_task() 行 556
       ├─ 生成 UUID task_id
       ├─ 内存写入 _TASKS dict
       └─ 持久化 JSON + MySQL
4. 推入 Worker 队列
   └─ _enqueue_task(task_id) 行 754-764
       ├─ _ensure_worker_started() 行 737-751：确保 daemon 线程已启动
       └─ _TASK_QUEUE.put(task_id) 行 764
5. 立即返回 task_id 给前端
   └─ 行 866
```

**Step 2 - 后台 Worker 执行**：

Worker 循环 `_worker_loop`（行 720-734）
```
queue.Queue.get() → _process_task(task_id)
```

### 3.3 研报生成核心函数：`_generate_report_markdown`（行 431-626）

> **严格 LLM 模式**，`AI_QUANT_REPORT_USE_LLM=1` 时才执行；否则抛异常

```
1. 前置检查
   ├─ AI_QUANT_REPORT_USE_LLM 环境变量 行 462
   └─ DASHSCOPE_API_KEY 行 469
2. MySQL 读取行情
   ├─ trade_stock_daily 最近 60 行 行 493-497
   └─ trade_financial 最新报告期 行 499-500
   └─ trade_stock_news 最近 30 条 行 502-506
3. RAG 检索（可选）
   └─ rag.py:rag_query() 行 524
       └─ 检索 "研报 财报 经营 风险" 相关段落（top-6）
4. 组装 system_prompt
   └─ 行 593-597：设定角色为"资深买方研究员"
5. 组装 user_prompt
   ├─ 日线行情表格（Markdown）行 572-577
   ├─ 新闻快照 行 580-590
   └─ RAG 材料 行 541
6. 调用 DashScope LLM
   └─ _dashscope_generate() 行 625
       ├─ threading.Thread 执行（daemon）行 276-278
       ├─ 超时 90s（AI_QUANT_REPORT_LLM_TIMEOUT_SECONDS）行 250-254
       └─ 响应解析同时兼容 output.text 和 output.choices 两种格式 行 292-304
7. 返回 Markdown 文本
```

### 3.4 内置模板兜底：`_builtin_report_markdown`（行 308-428）

> 当 `AI_QUANT_REPORT_USE_LLM` 非 1 时触发；通常在 LLM 不可用时生成骨架研报

```
1. 从 MySQL trade_stock_daily 读最近 30 行
2. 生成带涨跌的 Markdown 表格
3. 组装 6 个章节骨架（公司业务/财务/技术面/事件催化/风险清单/结论建议）
```

### 3.5 后台 Worker 处理：`_process_task`（行 651-717）

```
1. get_task(task_id) 行 666
2. update_task(status="running") 行 669
3. 对每只股票调用 _generate_report_markdown 行 674-675
4. 多只股票用分隔线拼接 行 676
5. 写入报告文件（行 679-690）
   ├─ 目标目录：AI_QUANT_REPORT_OUTPUT_DIR（权限检测 + PermissionError 回退到 .ai_quant/report_outputs/）
   ├─ 原子写入：先写 .tmp 再 rename
   └─ 失败自动回退默认目录 行 686-690
6. update_task(status="success") + report_markdown 字段 行 698
7. 异常捕获分支 行 699-717
   ├─ SystemExit → 友好提示"请检查 DASHSCOPE_API_KEY/索引配置"
   ├─ 提取 traceback 最后帧作为 error_location
   └─ update_task(status="failed")
```

## 五、研报内容结构（RAG + LLM 生成）

### 5.1 Prompt 设计

**system_prompt**（行 593-597）
```
你是资深买方研究员与投资经理助理。
请基于用户提供的数据快照与 RAG 材料，生成严谨、结构化、可读性强的中文个股研报。
必须输出 Markdown，包含所有章节；不要输出与任务无关的解释。
```

**user_prompt 组装顺序**（行 600-623）
```
1. 格式要求（一级标题顺序）
2. 日线行情表格（Markdown |...| 格式，限 20 行）
3. 新闻快照（限 10 条）
4. RAG 检索材料（限 6 条 chunk，每条含标题+正文）
5. 输出要求：直接输出 Markdown 正文
```

**LLM 兼容格式**（行 292-304）
```python
# 兼容新版 output.text
out.get("output", {}).get("text")
# 兼容旧版 output.choices[0].message.content
out.get("output", {}).get("choices", [{}])[0].get("message", {}).get("content")
```
