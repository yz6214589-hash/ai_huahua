## 一、项目背景与目标

### 1.1 项目背景

AI 量化交易系统（ai_quant）是一个功能完善的前端 Web 应用 + 后端 API 服务系统，包含以下 11 个核心业务模块：

| 序号 | 模块名称 | 前端页面 | 后端 API | 描述 |
|------|---------|---------|---------|------|
| 1 | 首页总览 | Dashboard.tsx | summary.py | 系统概览、数据统计 |
| 2 | 智能研报 | Reports.tsx | reports.py | 研报生成、RAG 检索、LLM 调用 |
| 3 | 数据与交付 | Data.tsx | data_charles.py | 数据查询、股票搜索 |
| 4 | 采集任务 | Jobs.tsx | jobs.py | 定时任务调度、数据采集管理 |
| 5 | 舆情监控 | Sentiment.tsx | sentiment.py | 舆情扫描、宏观指标 |
| 6 | 晨会简报 | Morning.tsx | console_ceo.py | 晨会分析、板块排名、股票筛选 |
| 7 | 风控中心 | Risk.tsx | risk_kris.py | 订单审批、风险检查、审计日志 |
| 8 | 执行监控 | Execution.tsx | execution_ethan.py | 交易执行、订单管理 |
| 9 | 自选股 | Watchlist.tsx | watchlist.py | 自选股管理 |
| 10 | 策略分析 | Strategy.tsx | analysis_zoe.py | 技术分析、信号识别 |
| 11 | AI 对话 | Chat.tsx | agent.py | AI 智能体、多轮对话 |

### 1.2 当前日志问题

1. **日志分散**：每个模块有各自的日志记录方式，缺乏统一管理
2. **格式不统一**：日志格式各异，难以解析和分析
3. **存储不规范**：日志存储位置混乱，难以定位和查找
4. **功能缺失**：缺少日志级别控制、轮转、查询等企业级功能
5. **调试困难**：出现问题时难以快速定位和追踪

### 1.3 设计目标

为 ai_quant 系统设计并实现一个统一、集中、规范的日志系统，实现以下目标：

1. **统一管理**：集中管理所有日志输出，消除日志分散问题
2. **格式规范**：统一日志格式，便于解析和分析
3. **分级控制**：支持多级别日志（DEBUG、INFO、WARNING、ERROR、CRITICAL）
4. **多维度分类**：按模块、功能、时间、级别等多维度组织日志
5. **自动轮转**：支持日志文件自动轮转，防止占用过多磁盘空间
6. **快速查询**：提供 API 接口和查询功能，便于快速查找日志
7. **敏感保护**：对敏感信息进行脱敏处理，保护数据安全
8. **便于追踪**：记录完整的请求链路，便于问题追踪和排查

#### 2.2.2 日志格式化器（Formatter）

**功能职责**：
- 统一日志输出格式
- 包含完整的时间戳、模块名、级别、内容
- 支持结构化日志输出

**格式定义**：

```
[2026-05-11 10:30:45] [reports] [INFO] 任务开始执行 task_id=abc123
[2026-05-11 10:30:45] [jobs] [ERROR] 数据库连接失败 host=127.0.0.1
[2026-05-11 10:30:46] [sentiment] [WARNING] 舆情采集中断 stock_code=600000
```

**格式字段**：
- `[timestamp]`: 时间戳，精确到秒
- `[module]`: 模块名称
- `[level]`: 日志级别
- `[message]`: 日志内容（支持结构化参数）

#### 2.2.3 敏感信息脱敏器（Sanitizer）

**功能职责**：
- 自动识别和脱敏敏感信息
- 支持自定义敏感字段
- 保护用户隐私和数据安全

**脱敏规则**：

| 敏感类型 | 识别模式 | 脱敏方式 |
|---------|---------|---------|
| API Key | `api_key=xxx`, `DASHSCOPE_API_KEY` | 保留前4位 + `****` |
| 密码 | `password=xxx` | 全部替换为 `******` |
| 手机号 | `1[3-9]\d{9}` | 保留前3后4位 |
| 身份证 | `\d{17}[\dXx]` | 保留前6后4位 |

#### 2.2.4 日志轮转管理器（Rotation Manager）

**功能职责**：
- 管理日志文件轮转
- 按大小或时间自动轮转
- 清理过期日志文件
- 防止磁盘空间占用过大

**轮转策略**：
- **大小轮转**：单个文件达到 10MB 时自动轮转
- **时间轮转**：每天凌晨 0 点自动轮转
- **备份保留**：保留最近 5 个备份文件
- **总大小限制**：日志目录总大小不超过 500MB

### 3.3 日志级别

## 五、安全性设计

2. **IP 白名单**：可配置允许访问日志的 IP 列表

### 5.3 日志完整性

1. **不可篡改**：日志文件使用追加模式，防止篡改
2. **完整性检查**：支持日志文件哈希校验
3. **审计追踪**：记录日志查询和访问行为

## 六、性能设计

### 6.1 性能目标

- 日志记录延迟 < 1ms
- 高并发下日志吞吐量 > 10000 条/秒
- 磁盘 I/O 不影响业务处理

### 6.2 优化策略

1. **异步写入**：使用队列缓冲，非阻塞写入
2. **批量写入**：累积多条日志后批量写入
3. **内存缓冲**：使用内存缓冲区减少磁盘 I/O
4. **分级存储**：错误日志实时写入，普通日志异步写入

### 6.3 资源限制

1. **磁盘空间**：日志目录总大小不超过 500MB
2. **文件数量**：单个日志类别最多 10 个文件
3. **保留时间**：默认保留 7 天

## 九、风险与应对

### 9.1 风险识别

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 日志影响性能 | 高 | 中 | 使用异步写入、批量处理 |
| 磁盘空间不足 | 中 | 低 | 监控告警、自动清理 |
| 日志泄露敏感信息 | 高 | 低 | 脱敏处理、访问控制 |
| 日志格式不统一 | 中 | 中 | 统一规范、代码审查 |
| 轮转策略失效 | 中 | 低 | 监控检查、手动触发 |

### 9.2 应急预案

1. **磁盘空间不足**：立即清理旧日志，通知运维
2. **日志无法写入**：切换到控制台输出，排查问题
3. **敏感信息泄露**：立即停止系统，排查泄露原因

## 十、总结

本文档详细设计了 ai_quant 量化交易系统的日志系统架构，包括：

1. **统一管理**：集中管理所有日志，消除分散问题
2. **规范格式**：统一日志格式，便于解析和分析
3. **分级控制**：支持多级别日志，满足不同场景需求
4. **多维分类**：按模块、功能、时间、级别等多维度组织
5. **自动轮转**：防止磁盘空间占用过大
6. **快速查询**：提供 API 接口，便于查找和分析
7. **敏感保护**：自动脱敏敏感信息，保护数据安全
8. **便于追踪**：记录完整请求链路，便于问题排查

该日志系统将显著提升系统的可观测性、可维护性和安全性，为后续的运维和问题排查提供有力支持。
