# 团队评审 Checklist（交付物 100% 达标口径）

本清单用于评审“`.ai_quant` 运行时目录技术交底”交付物是否完备、可复现、可运行。评审通过标准：所有条目均满足。

## A. 交底文档完整性

- [ ] 文档已覆盖技术路线综述：选型理由、演进规划、风险评估
- [ ] 文档已覆盖前后端技术栈明细：语言、框架、中间件、版本锁定策略、CI/CD 工具链（含现状与规划）
- [ ] 文档已覆盖数据库设计：逻辑模型、物理模型、容量估算、索引策略、分库分表规则、备份与容灾方案
- [ ] 文档已覆盖信息架构：用户旅程、功能地图、导航结构、权限矩阵
- [ ] 文档已覆盖接口文档：REST + GraphQL（Schema 源文件）、请求/响应模型、错误码、鉴权流程、Mock 规则、变更记录
- [ ] 文档已覆盖前端交互：页面流程图、组件状态图、可访问性标准、性能指标、埋点方案
- [ ] 文档中所有敏感信息均已脱敏（`***`），不包含真实 Key/密码/Token

## B. `.ai_quant` 运行时目录交付质量

- [ ] 已明确 `.ai_quant` 的职责边界与生命周期（可清理、可重建、可挂载）
- [ ] 已给出 `.ai_quant` 规范结构，并与代码默认路径一致
- [ ] 已明确哪些模块会写入 `.ai_quant`：研报日志、研报产物、RAG 存储、索引文件
- [ ] 已指出路径不一致的点（例如 job_runs），并给出统一策略（通过 env/Compose 归一）

## C. 图源与图例一致性（UML 2.5 规范）

- [ ] PlantUML 图源可编辑、可渲染，且均引用统一样式文件 `_style.puml`
- [ ] 图例包含：角色/颜色/关系线型说明（UML 2.5 口径）
- [ ] 颜色与图标（形状/构件）在所有图中一致
- [ ] 分层/组件/部署/安全 4 类架构图齐全
- [ ] 信息架构图齐全：用户旅程、功能地图、导航结构、权限矩阵
- [ ] 前端交互图齐全：页面流程图、组件状态图
- [ ] 数据库 ER 图齐全：MySQL + SQLite（RAG）
- [ ] DrawIO 源文件存在且可编辑（至少覆盖 4 类架构视图）

## D. Docker Compose 一键验证环境

- [ ] `docker compose up --build` 可在本机成功启动（无需手工安装 Python/Node 依赖）
- [ ] MySQL 自动初始化最小 schema（init.sql 生效）
- [ ] 后端 `GET /api/health` 返回 200
- [ ] Web 可访问并能正常调用后端（/api 代理正常）
- [ ] Streamlit 可访问并能调用后端 API
- [ ] `.ai_quant` 作为 volume 挂载，容器运行后日志与产物可在宿主机看到

## E. 关键业务验收（最小闭环）

- [ ] 研报任务创建后不会“创建即失败”（除缺少配置/索引等明确错误）
- [ ] 研报生成成功后，产物落盘到 `.ai_quant/report_outputs/<task_id>.md`
- [ ] 研报 worker 日志持续追加到 `.ai_quant/reports_worker.log`
- [ ] RAG ingest/build/query 三个接口可用（或明确标注需要准备 PDF/Key 的前置条件）
