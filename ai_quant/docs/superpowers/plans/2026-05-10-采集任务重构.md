# 采集任务（Jobs）重构实现计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**目标：**将 `/api/jobs/run`、`/api/jobs/runs`、`/api/jobs/schedules` 从“仅写运行记录”升级为“可真实执行采集任务 + 运行记录落盘 + 调度配置持久化 + 可选定时调度”，并把参考系统（charles）的 jobs 能力适配到当前 `ai_quant` 的后端架构与前端风格。

**架构：**后端保持 FastAPI + Router 结构；新增 jobs 执行器（按 domain 分派）与 JobRun JSON 存储；调度配置落 MySQL 表（自动建表），并用 APScheduler 触发后台执行；前端尽量复用现有 Jobs 页面与类型，仅做必要的参数对齐与展示增强。

**技术栈：**FastAPI、Pydantic v2、PyMySQL、APScheduler（新增）、AkShare+Pandas（新增，用于采集实现）、OpenAI 兼容接口（可选，催化剂/新闻摘要用，按环境变量启用）。

---

## 文件结构（锁定边界）

**后端（ai_quant/backend）**
- 新增：`ai_quant_api/services/charles/jobs/`：各 domain 的采集实现（参考 charles/api/charles_api/jobs/*，适配当前表结构）
- 新增：`ai_quant_api/services/charles/jobs/common.py`：JobStats、参数解析与通用工具
- 新增：`ai_quant_api/services/charles/jobs/runner.py`：domain -> run_* 分派与统一异常/状态落盘
- 修改：`ai_quant_api/api/jobs.py`：改造成参考系统的 API 行为（run/runs/schedules），并接入后台执行
- 修改：`ai_quant_api/app.py`：注册 scheduler 的 startup/shutdown（如启用 APScheduler）
- 修改：`backend/requirements.txt`：补齐采集与调度依赖（akshare、pandas、apscheduler、pyyaml、openai）
- 可选新增：`docker/mysql/init.sql` 补齐 `trade_job_schedule`（或者仅在后端启动时自动建表）

**前端（ai_quant/web）**
- 修改：`src/pages/Jobs.tsx`：由后端模型驱动入参（移除硬编码 test_stock 的默认值，改为可编辑/可配置）
- 修改：`src/api/types.ts`：与后端返回字段对齐（如 userMessage）
